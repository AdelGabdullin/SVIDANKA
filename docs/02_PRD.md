## 1. Обзор

Документ детализирует [`01_Product_Vision.md`](01_Product_Vision.md) до уровня, достаточного для постановки задач в GitHub Issues и проектирования (Domain Model, Database Design, API Specification).

PRD фиксирует: функциональные и нефункциональные требования MVP, роли пользователей, ограничения проекта, бизнес-правила предметной области, пользовательские сценарии и критерии приёмки. Следующие документы в цепочке — `03_User_Stories.md` и `04_Use_Cases.md` — детализируют требования этого PRD до конкретных историй и сценариев использования.

## 2. Функциональные требования

### 2.1 Города

- FR-1. Пользователь выбирает город из списка при входе в сервис.
- FR-2. На старте (MVP) поддерживается несколько крупных городов России (города-миллионники); список городов расширяется администратором без изменения кода.

### 2.2 Карта и список мест

- FR-3. Места отображаются на интерактивной карте (Leaflet, см. ADR-003) и в виде списка одновременно.
- FR-4. Каждое место на карте — кликабельный маркер, ведущий на карточку места.

### 2.3 Карточка места

- FR-5. Карточка места содержит: фотографии, описание, адрес, категории, стоимость (бюджет), рейтинг, отзывы.

### 2.4 Категории

- FR-6. Список категорий мест — открытый и управляется администратором через админ-панель (не хардкодится в коде). Примеры: кафе/рестораны, парки/прогулки, активный отдых, культура, смотровые площадки, развлечения.

### 2.5 Фильтрация

- FR-7. Доступны фильтры: город, бюджет (диапазон), район, расстояние, категория.
- FR-8. Фильтры комбинируются между собой (логическое И).
- FR-9. Бюджет задаётся целыми числами в рублях, без дробной части, диапазон от 0 до 100 000 ₽, отрицательные значения не допускаются.

### 2.6 Режим «Кубик»

- FR-10. По кнопке «Кубик» сервис случайным образом выбирает одно место из набора мест, отфильтрованных активными фильтрами.
- FR-11. Если под заданные фильтры не подходит ни одно место, пользователю показывается сообщение об отсутствии подходящих мест (а не ошибка).

### 2.7 Отзывы

- FR-12. Любой пользователь может анонимно оставить отзыв к месту: оценка (1–5) + текстовый комментарий.
- FR-13. Отзывы отображаются в карточке места сразу после отправки (постмодерация, см. раздел 6.1).
- FR-14. В MVP нет ограничений на количество отзывов от одного человека (регистрации нет, риск спама принимается на уровне постмодерации; см. раздел 5).

### 2.8 Рейтинг

- FR-15. Рейтинг места рассчитывается как среднее арифметическое оценок всех его отзывов.
- FR-16. Если у места нет отзывов, рейтинг не отображается (вместо числа — пометка «нет оценок»).

### 2.9 Предложение нового места

- FR-17. Любой пользователь может предложить новое место через форму (название, адрес, категория, бюджет, описание, фотографии).
- FR-18. Предложенное место публикуется сразу (постмодерация) и становится видимым всем пользователям.

### 2.10 Администрирование и модерация

- FR-19. Администратор может добавлять места напрямую, минуя форму предложения.
- FR-20. Администратор может скрыть/удалить любое место или отзыв постфактум, если они не соответствуют требованиям сервиса.
- FR-21. Администратор управляет справочниками городов и категорий через админ-панель.

## 3. Нефункциональные требования

### 3.1 Производительность

- NFR-1. Ответ API для чтения данных (список мест, карточка места, фильтрация) — не более 300 мс (p95) без учёта времени загрузки тайлов карты.
- NFR-2. Полная загрузка страницы со списком мест и картой — не более 2 секунд на стабильном соединении.
- NFR-3. Ответ «Кубика» — не более 1 секунды после нажатия кнопки.

### 3.2 Масштабируемость

- NFR-4. Архитектура должна выдерживать одновременную работу нескольких городов-миллионников без деградации производительности при добавлении новых городов/мест (места — тысячи записей на город).
- NFR-5. Добавление нового города или категории не требует изменения кода (только данные через админ-панель).

### 3.3 Безопасность

- NFR-6. Персональные данные не собираются (см. Product Vision, раздел 12) — снижает требования к защите ПДн.
- NFR-7. Жёстких лимитов на частоту отзывов/предложений от одного IP/устройства в MVP нет (осознанное решение, риск спама компенсируется постмодерацией администратором).
- NFR-8. Весь пользовательский контент (отзывы, предложенные места) проходит через возможность постфактум-удаления администратором.

### 3.4 Доступность и мониторинг

- NFR-9. Целевая доступность сервиса на этапе MVP — best-effort (без формального SLA), с намерением выйти на 99% по мере роста нагрузки.
- NFR-10. Мониторинг реализуется через Prometheus + Grafana (см. стек в `AGENTS.md`): отслеживаются задержка API, частота ошибок, состояние БД/очередей.

## 4. Роли пользователей

### 4.1 Гость (аноним)

Основная роль в MVP, регистрация не требуется. Может:

- просматривать карту, список мест и карточки;
- фильтровать места и пользоваться «Кубиком»;
- оставлять анонимные отзывы (он же «автор отзыва»);
- предлагать новые места (он же «автор предложенного места»).

### 4.2 Администратор (он же модератор)

Единая роль на первое время. Может всё, что доступно гостю, а также:

- добавлять места напрямую, минуя модерацию;
- скрывать/удалять неподходящие места и отзывы постфактум;
- управлять справочниками городов и категорий.

Разделение на отдельные роли «Администратор» и «Модератор» — за пределами MVP (см. раздел 9).

## 5. Ограничения

- Технологические: используется только зафиксированный стек (Python 3.13, FastAPI, PostgreSQL, SQLAlchemy, Alembic, Redis, RabbitMQ, React + TypeScript, Docker, Leaflet). Новые технологии не вводятся без обсуждения и записи ADR (см. `AGENTS.md`).
- Команда: над проектом работают два человека; требования и объём MVP должны оставаться реалистичными для этого формата.
- Сроки: жёсткого дедлайна запуска не зафиксировано; ориентир — последовательное прохождение этапов Discovery → Проектирование → Backend → Frontend → QA → DevOps (см. `AGENTS.md`, раздел «Жизненный цикл»).
- Бюджет/инфраструктура: используются бесплатные/условно-бесплатные компоненты (Leaflet + OpenStreetMap вместо Google Maps, см. ADR-003), без выделенного инфраструктурного бюджета на старте.
- Регуляторные: отсутствие сбора персональных данных в MVP снимает необходимость соответствия требованиям обработки ПДн на раннем этапе.

## 6. Бизнес-правила

### 6.1 Модерация

- Действует постмодерация: и места, и отзывы публикуются сразу после отправки, без предварительной проверки.
- Администратор вправе в любой момент скрыть или удалить место/отзыв, не соответствующие тематике сервиса (не про свидания/отдых), содержащие оскорбления, спам или недостоверную информацию.

### 6.2 Отзывы

- Отзыв = оценка (1–5) + текстовый комментарий, публикуется анонимно.
- Ограничений на количество отзывов от одного человека в MVP нет.

### 6.3 Рейтинг

- Рейтинг места = среднее арифметическое всех его оценок.
- При отсутствии отзывов рейтинг не показывается.

### 6.4 Фильтрация и «Кубик»

- Фильтры комбинируются по И (город + бюджет + район + расстояние + категория).
- «Кубик» выбирает случайное место только из мест, прошедших текущие фильтры.
- Бюджет — целое неотрицательное число в рублях, не более 100 000 ₽.

### 6.5 Ограничения MVP

- Нет регистрации пользователей.
- Нет сбора персональных данных.
- Отзывы и предложения мест анонимны.
- Категории и города — открытые списки, управляемые администратором.

## 7. Пользовательские сценарии

### 7.1 Основной сценарий — подбор места

```text
Открыть сайт → Выбрать город → Указать бюджет → Выбрать категорию →
Получить список мест на карте → Открыть карточку → Поехать →
Вернуться и оставить отзыв
```

### 7.2 Сценарий «Кубик»

```text
Открыть сайт → Выбрать город → Настроить фильтры (по желанию) →
Нажать «Кубик» → Получить случайное место из отфильтрованных →
Посмотреть карточку → Поехать
```

Если по заданным фильтрам мест нет — пользователь видит сообщение об отсутствии подходящих вариантов и может изменить фильтры.

### 7.3 Сценарий предложения нового места

```text
Найти место, которого нет в базе → Заполнить форму предложения
(название, адрес, категория, бюджет, описание, фото) → Отправить →
Место сразу публикуется (постмодерация) → Администратор при необходимости
скрывает/удаляет место позже
```

### 7.4 Сценарий работы администратора

```text
Открыть админ-панель → Просмотреть новые места/отзывы →
При необходимости скрыть/удалить неподходящий контент →
Добавить место напрямую (минуя форму предложения) →
Управлять справочником городов/категорий
```

## 8. Критерии приёмки

### 8.1 Ключевые сценарии (Given-When-Then)

**Кубик учитывает фильтры**
```gherkin
Given пользователь выбрал город и задал фильтры (бюджет, категория)
When пользователь нажимает «Кубик»
Then сервис возвращает одно место, соответствующее всем заданным фильтрам
```

**Кубик без подходящих мест**
```gherkin
Given заданные фильтры не соответствуют ни одному месту в выбранном городе
When пользователь нажимает «Кубик»
Then сервис показывает сообщение об отсутствии подходящих мест без ошибки
```

**Постмодерация отзыва**
```gherkin
Given пользователь отправил отзыв к месту
When отзыв сохранён
Then отзыв немедленно виден всем пользователям без предварительной проверки
```

**Удаление неподходящего контента администратором**
```gherkin
Given администратор находит место или отзыв, не соответствующие требованиям сервиса
When администратор скрывает/удаляет этот контент через админ-панель
Then контент перестаёт отображаться для остальных пользователей
```

### 8.2 Чек-лист по остальным фичам MVP

- [ ] Пользователь может выбрать город из списка.
- [ ] Карта отображает места выбранного города с кликабельными маркерами.
- [ ] Список мест синхронизирован с картой и фильтрами.
- [ ] Карточка места отображает фото, описание, адрес, бюджет, категории, рейтинг, отзывы.
- [ ] Фильтры (бюджет, район, расстояние, категория) работают в комбинации.
- [ ] Бюджет вводится и отображается как целое число в рублях (0–100 000).
- [ ] Пользователь может анонимно оставить отзыв (оценка + текст).
- [ ] Рейтинг пересчитывается как среднее оценок; при отсутствии отзывов — «нет оценок».
- [ ] Пользователь может предложить новое место через форму.
- [ ] Администратор может добавлять места напрямую и управлять справочниками городов/категорий.
- [ ] Регистрация отсутствует, персональные данные не собираются.

## 9. Будущие версии

Не входит в MVP, планируется после запуска (детали — в `01_Product_Vision.md`, раздел 13, и в `12_Roadmap.md`):

- Пользовательские аккаунты (избранное, история, персональные рекомендации, достижения).
- Разделение единой роли «Администратор/Модератор» на отдельные роли с разными правами.
- Более сложный алгоритм рейтинга (с учётом количества отзывов и их свежести).
- Лимиты и антиспам-механизмы (rate limiting, капча) при добавлении отзывов/мест, если постмодерация перестанет справляться с нагрузкой.
- Построение маршрутов / генератор сценариев свидания (например: Кофейня → Парк → Мороженое → Смотровая площадка).
- Расширение географии на всю Россию.
- Коммерческие аккаунты для заведений (официальные страницы, реклама, продвижение, спецпредложения).
