## 1. Идея продукта

**Свиданка** — веб-платформа для быстрого поиска интересных мест для свиданий, прогулок и совместного отдыха.

Пользователь выбирает город, бюджет и категорию — и получает подходящие места на карте или через режим случайного выбора («Кубик»).

Главная цель — сделать выбор места таким же простым, как заказ такси.

---

## 2. Проблема

Идеи мест сегодня разбросаны по десяткам источников:

- сохранённые публикации Instagram;
- Telegram-каналы;
- TikTok, YouTube Shorts;
- Яндекс Карты, 2ГИС;
- заметки в телефоне;
- рекомендации друзей.

В результате:

- информация теряется и устаревает;
- сложно быстро найти подходящий вариант;
- нет единой базы именно **мест для свиданий**;
- нельзя выбрать место случайно, если нет конкретных предпочтений.

Поиск места нередко занимает больше времени, чем сама встреча.

---

## 3. Решение

Веб-платформа, которая позволяет за пару минут подобрать место. Пользователь:

1. выбирает город;
2. указывает бюджет;
3. применяет дополнительные фильтры (район, расстояние, категория);
4. получает карту и список подходящих мест;
5. либо жмёт **«🎲 Кубик»** — сервис предлагает случайное место под заданные фильтры.

---

## 4. Целевая аудитория

**Основная:** мужчины и женщины 18–40 лет, пары, люди перед первым свиданием, жители крупных городов России.

**Вторичная:** туристы, путешественники, новые жители города, компании друзей, семейные пары.

---

## 5. MVP

Первая версия включает:

- **Карта:** отображение мест, переход в карточку.
- **Города:** выбор города.
- **Фильтрация:** бюджет, район, расстояние, категория.
- **Карточка места:** фотографии, описание, адрес, стоимость, категории, рейтинг.
- **Отзывы:** просмотр и добавление анонимного отзыва.
- **Random Mode:** кнопка «🎲 Кубик» — случайное место под фильтры.
- **Наполнение базы:** пользователи предлагают места → модерация → публикация.

## 7. Персональные данные

На этапе MVP регистрация **отсутствует**. Сервис принципиально не собирает: имя, фамилию, телефон, email, паспортные данные, постоянную геолокацию. Отзывы публикуются анонимно.

Это снижает порог входа и убирает необходимость обработки персональных данных на раннем этапе.

---

## 8. Уникальные особенности

- **🎲 Random Date.** Пользователь доверяет выбор сервису — алгоритм подбирает место с учётом бюджета, города, категории и расстояния. Превращает выбор в элемент игры.
- **Простота.** Открыл сайт — сразу пользуешься. Без регистрации, подтверждения телефона, заполнения профиля.
- **Сообщество.** Любой может предложить новое место; после модерации оно попадает в каталог.

---

## 9. Долгосрочное развитие

- **Аккаунты:** избранное, история посещений, достижения, персональные рекомендации.
- **Система рейтингов:** автоматический рейтинг на основе оценок, количества отзывов, актуальности и активности.
- **Маршруты / генератор сценариев свидания** *(потенциально главная фишка)*: сервис отдаёт не одно место, а готовый вечер. Например:
  `☕ Кофейня → 🌳 Парк → 🍨 Мороженое → 🌇 Смотровая площадка`.
- **Карта всей России.**
- **Коммерческие аккаунты:** официальные страницы заведений, реклама, продвижение карточек, спецпредложения.

---

## 10. Критерии успеха MVP

Продукт успешен, если пользователь может:

- подобрать место менее чем за 2 минуты;
- найти локацию без сторонних сервисов;
- воспользоваться случайным выбором;
- предложить новое место;
- оставить отзыв.

---

## 11. Ключевая продуктовая гипотеза (обсуждение)

Сейчас формулировка звучит как **«каталог мест для свиданий»** — полезно, но таких каталогов много.

Если сместить фокус на **«генератор сценариев свиданий»**, продукт становится сильнее: пользователь выбирает бюджет / время / транспорт / настроение и получает **готовый вечер**, а не просто точку на карте. Именно это может выделить проект среди аналогов.

---

## 12. Подход к разработке

Проект ведётся как настоящий коммерческий продукт, а не «pet-проект для галочки»: документация **до** кода, осознанные архитектурные решения с их фиксацией, тесты на API и UI, GitHub Projects / Issues, Docker, CI/CD, мониторинг.

### Жизненный цикл

**Этап 1. Discovery**
- ✅ Product Vision
- ⏳ PRD
- ⏳ User Stories
- ⏳ Use Cases
- ⏳ Business Rules

**Этап 2. Проектирование** — Domain Model, ER Diagram, Database Design, API Design (OpenAPI), C4 Architecture, Sequence Diagrams.

**Этап 3. Backend** — Python, FastAPI, PostgreSQL, SQLAlchemy, Alembic, Redis, RabbitMQ, Docker.

**Этап 4. Frontend** — React + TypeScript.

**Этап 5. QA** — тест-план, чек-листы, тест-кейсы, API-автотесты, UI-автотесты, нагрузочное тестирование, отчёты Allure.

**Этап 6. DevOps** — Docker Compose, GitHub Actions, Nginx, HTTPS, Prometheus, Grafana.

## 15. Decision Log

Фиксируем «почему», чтобы через полгода можно было открыть и вспомнить контекст.

| # | Решение | Обоснование |
|---|---------|-------------|
| D-1 | Без регистрации в MVP | Минимальный порог входа, нет обработки персональных данных |
| D-2 | Анонимные отзывы | То же — упрощение и снижение юридических требований |
| D-3 | Режим «Кубик» | Игровой элемент + снимает паралич выбора; потенциальная фишка |
| D-4 | Документация до кода | Проект как коммерческий продукт, сильный кейс на собеседовании |
| D-5 | Ведение ADR | Явная фиксация архитектурных решений — редкость в pet-проектах |

Планируемые ADR: `ADR-001` — стек / FastAPI, `ADR-002` — PostgreSQL, `ADR-003` — Leaflet вместо Google Maps, (доп.) RabbitMQ вместо Kafka.

## 16. Работа через VS Code / Codex

Текущий диалог не «переезжает» в VS Code автоматически — контекст надо передать файлами. План:

1. Сохранить этот документ как `docs/meetings/product-discovery-01.md`.
2. Открыть проект в VS Code.
3. Дать агенту инструкцию:
   ```
   Прочитай docs/meetings/product-discovery-01.md, AGENTS.md и PROJECT_CONTEXT.md.
   Ты работаешь над проектом Свиданка. Продолжи разработку согласно документации.
   Сейчас подготовь структуру backend-приложения.
   ```
4. Дальше — попросить выделить из документа главное и сгенерировать `PROJECT_CONTEXT.md`, `AGENTS.md`, `01_Product_Vision.md`.
