# ADR-005: Вход администратора без системы аккаунтов

- **Статус:** Accepted
- **Дата:** 2026-07-22
- **Автор:** Adel Gabdullin

## Контекст

В MVP нет регистрации и системы пользовательских аккаунтов (ADR-004). При этом нужен способ для администратора попадать в админ-панель (модерация, справочники, feature flags — см. `02_PRD.md`, разделы 2.10–2.12) без введения полноценной системы логинов/ролей на раннем этапе.

## Решение

Явной кнопки или ссылки «Войти» в публичном интерфейсе нет. Доступ к админ-панели скрыт: 5 быстрых кликов по логотипу открывают форму ввода пароля; успешный ввод пароля открывает админ-панель. Пароль общий на роль «Администратор» (см. `02_PRD.md`, раздел 4.2) — отдельных учётных записей для нескольких администраторов в MVP не заводится.

## Рассмотренные альтернативы

- Отдельная страница `/admin/login` с формой логина — технически проще, но напрямую выдаёт злоумышленнику точку входа для перебора пароля; требует также решить, что делать с полем «логин», которого при общем пароле на роль просто нет.
- Полноценная система аккаунтов с ролями (Admin/Moderator) и email+пароль — избыточно для одной текущей роли и одного-двух реальных администраторов; отложено до момента, когда роли разделятся (см. `02_PRD.md`, раздел 9 «Будущие версии»).
- OAuth через внешнего провайдера (Google и т.п.) — не нужен на этом масштабе, добавляет внешнюю зависимость ради одной учётной записи.

## Последствия

Плюсы: не нужно проектировать таблицу пользователей/ролей на этапе MVP, скрытая точка входа снижает шанс случайного/автоматического перебора пароля ботами, которые не знают о жесте активации. Минусы: security through obscurity — не замена полноценной защиты, а дополнительный слой (пароль всё ещё должен быть надёжным и не передаваться по незащищённым каналам); при заведении второго администратора с личными правами доступа потребуется полноценная система аккаунтов — уже заложено как пункт будущих версий.
