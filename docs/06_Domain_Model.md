## 2. City

Город, внутри которого пользователь ищет места. Управляется администратором (см. `04_Use_Cases.md`, UC-07; `05_Business_Rules.md`, BR-15).

**Атрибуты:** id, name (название), slug, is_active (для деактивации без физического удаления).

**Связи:** 1 City → N Place.

## 3. Place

Центральная сущность — место для свидания/прогулки/отдыха.

**Атрибуты:** id, city_id, name, description (текст, максимум 1500 символов — BR-21), budget (целое число, 0–100 000 ₽ — BR-10), status (published / hidden / archived — см. раздел 9), avg_rating и reviews_count (кэш — см. раздел 10), added_by_admin (флаг: добавлено администратором напрямую — FR-19 — или через форму предложения — FR-17/FR-18), created_at.

**Локация (FR-17.1):** место указывается либо адресом (`address`, свободный текст), либо координатами (`latitude`, `longitude`) — оба поля опциональны на уровне схемы, но обязательно наличие хотя бы одного из вариантов на уровне бизнес-логики (проверяется в API, не в БД).

**Связи:** N Place → 1 City; N Place ↔ N Category; N Place ↔ N Tag (см. раздел 8); 1 Place → N Photo (0–3); 1 Place → N Review.

## 4. Category

Категория места. Открытый список, управляется администратором (BR-15).

**Атрибуты:** id, name, slug, is_active.

**Связи:** N Category ↔ N Place — у одного места может быть **несколько категорий** одновременно (например, «Кафе» и «Смотровая площадка»).

## 5. Review

Анонимный отзыв на место (FR-12–FR-14; BR-04–BR-06).

**Атрибуты:** id, place_id, rating (целое, 1–5), comment (максимум 1500 символов — BR-05/BR-21), status (published / hidden — см. раздел 9), created_at.

**Связи:** N Review → 1 Place.

## 6. Photo

Фотография места (BR-22).

**Атрибуты:** id, place_id, url/path, order (порядок отображения), created_at.

**Ограничение:** место может быть опубликовано без фотографий; если фотографии есть — не более 3 на место. Ограничение проверяется на уровне сервиса (backend), не на уровне схемы БД.

**Связи:** N Photo → 1 Place.

## 7. Route

*Статус: пост-MVP.* Маршрут/сценарий свидания из нескольких мест по порядку (например: Кофейня → Парк → Мороженое → Смотровая площадка — `01_Product_Vision.md`, раздел 13). Не используется в MVP UI/API, но модель заложена, чтобы не переделывать схему при переходе к этой фиче.

**Атрибуты:** id, name, description, created_at.

**Связи:** 1 Route → N RouteStep; N RouteStep → 1 Place (RouteStep хранит порядковый номер места в маршруте).

## 8. Tag

*Статус: заложено на будущее, не входит в текущие FR PRD.* Свободный тег места, в дополнение к фиксированным категориям (например, «с видом на закат», «работает допоздна»). Если будет использоваться в MVP-интерфейсе — потребуется отдельное FR в `02_PRD.md`.

**Атрибуты:** id, name, slug.

**Связи:** N Tag ↔ N Place.

## 9. Moderation

*Не отдельная сущность.* Модерация в MVP реализована как **статус на Place и Review** (BR-01–BR-03, BR-16–BR-18):

- `published` — видно всем пользователям (состояние по умолчанию сразу после создания — постмодерация).
- `hidden` — скрыто администратором вручную (место/отзыв не по теме, спам, оскорбления и т.п.); может быть восстановлено обратно в `published`.
- `archived` — место, у которого администратор удалил/деактивировал город или категорию (BR-16); видно только администратору. Из архива администратор может либо безвозвратно удалить место, либо вручную вернуть его в `published` (разархивировать) на своё усмотрение — см. BR-17.

Отдельная сущность-журнал модерации (кто/когда скрыл/восстановил) в MVP не заводится — это осознанное упрощение, к которому можно вернуться в будущих версиях, если понадобится аудит действий администратора.

## 10. PlaceRating

*Не отдельная сущность.* Рейтинг места кэшируется прямо на Place (поля `avg_rating`, `reviews_count` — см. раздел 3), а не хранится отдельной таблицей и не считается на лету при каждом запросе. Пересчитывается при создании, изменении статуса или удалении отзыва (BR-07, BR-08).

## 11. RandomRecommendation

*Статус: заложено на будущее.* Лог отдельных выборов «Кубика» — не требуется функциональными требованиями MVP (FR-10/FR-11 не предполагают логирования), но модель заложена для будущей аналитики/персонализации рекомендаций.

**Атрибуты:** id, place_id, filters_snapshot (город/бюджет/категория/район/расстояние на момент выбора), created_at.

**Связи:** N RandomRecommendation → 1 Place.

## 11.1 FeatureFlag

Переключатель функционального блока сайта, управляется администратором (FR-22, FR-23; BR-24, BR-25; `13_UI_UX_Design.md`, раздел 12). Не связан ни с одной другой сущностью напрямую — влияет на поведение API/UI по ключу флага.

**Атрибуты:** id, key (уникальный технический ключ, например `budget_filter`, `cube_mode`), name (человекочитаемое название), enabled (boolean), updated_at.

**Примеры флагов MVP:** фильтр по бюджету, фильтр по району, фильтр по расстоянию, фильтр по категориям, режим «Кубик», отзывы, предложение мест гостями.

## 12. Связи между сущностями

```text
City (1) ──── (N) Place
Place (N) ──── (N) Category
Place (N) ──── (N) Tag                      [заложено на будущее]
Place (1) ──── (N) Photo                     [0..3]
Place (1) ──── (N) Review
Place (1) ──── (N) RandomRecommendation       [заложено на будущее]
Route (1) ──── (N) RouteStep ──── (1) Place   [пост-MVP]
```

Moderation и PlaceRating не имеют собственных таблиц — реализованы как атрибуты (`status`, `avg_rating`, `reviews_count`) на Place/Review. FeatureFlag — отдельная независимая таблица без связей с другими сущностями. Подробная схема с типами полей и ограничениями — в `07_Database_Design.md`.
