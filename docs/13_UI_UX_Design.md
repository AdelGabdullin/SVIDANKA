> Документ отвечает на вопрос: **Как продукт должен выглядеть и ощущаться, и как это подтверждено прототипом?**

Документ фиксирует визуальное направление, провалидированное в кликабельном HTML-прототипе ([`docs/mockups/svidanka_mockup.html`](mockups/svidanka_mockup.html) — черновой макет для согласования направления, открывается напрямую в браузере после клонирования репозитория). Основа для будущей реализации на React + TypeScript (`09_System_Architecture.md`, `frontend/`).

## 1. Общее направление

Тёмная тема с одним ярким акцентом — сочетание розово-малинового (`#ff2d78`) и оранжевого (`#ff7a3d`) в градиенте, на почти чёрном фоне (`#0a0a0f` / `#101018`). Направление осознанно смелее нейтрального SaaS-стиля: у продукта одна ключевая эмоция (игра, скорость выбора), и акцентный цвет держит это ощущение на каждом экране, а не только на кнопках.

## 2. Типографика

- Заголовки — **Unbounded** (500–800), геометричный и объёмный шрифт, держит смелый тон бренда.
- Основной текст, кнопки, лейблы — **Plus Jakarta Sans** (400–800), нейтральный и читаемый в паре с ярким Unbounded.
- Оба шрифта — Google Fonts, подключаются через `<link>`, лицензионных ограничений нет.

## 3. Цвета

| Роль | Значение | Применение |
|------|----------|------------|
| Фон | `#0a0a0f` → `#17111e` (радиальный градиент) | Страница |
| Поверхность карточки | `#15151f` | Карточки мест, модалки, панели |
| Акцент 1 | `#ff2d78` | Основной цвет CTA, свечения, статусов |
| Акцент 2 | `#ff7a3d` | Второй цвет в градиентах кнопок/логотипа |
| Текст основной | `#f5f5fa` | — |
| Текст вторичный | `#9a9aad` | Подписи, метаданные |

## 4. Карта

Не стандартная светлая Leaflet-подложка, а кастомная тёмная схема: точечная сетка (grid pattern) + радиальные акцентные пятна на фоне, кастомные маркеры-точки с пульсирующим кольцом (аналог "live"-индикатора) вместо стандартных pin-иконок. При клике на маркер — подпрыгивание + мини-вспышка искр. Технически на реальном фронтенде это достигается кастомным тайл-стилем Leaflet + собственными DivIcon-маркерами (см. ADR-003 — провайдер карт не меняется, меняется только оформление).

## 5. Карточка места

Крупное цветное изображение (в проде — фото места), пилюля категории и кнопка «избранное» поверх фото, ниже — название, рейтинг, дистанция/бюджет и короткий "цепляющий" тег-строка (не путать с бизнес-тегами из `06_Domain_Model.md` — здесь это просто маркетинговая подпись). При наведении — приподнятие карточки и акцентная обводка. Клик по карточке даёт тактильный "press"-отклик (лёгкое 3D-нажатие).

## 6. Фильтры: город и категории

Оба вынесены из общей панели фильтров в собственные раскрывающиеся блоки над картой:

- **Город** — одиночный выбор (радио-логика): выбор одного пункта сразу закрывает список и обновляет подпись на кнopke.
- **Категории** — множественный выбор: каждая выбранная категория помечается галочкой, счётчик выбранных показывается на самой кнопке-переключателе.

Список городов/категорий в этих выпадающих блоках — не статичный текст, а прямое отражение справочников, которыми управляет администратор (см. раздел 9 и `04_Use_Cases.md`, UC-07): деактивированный город или категория пропадают из списка на сайте.

## 7. Микроанимации

Каждый интерактивный элемент имеет собственную, отличимую от других реакцию — это осознанный принцип, а не украшательство ради украшательства:

| Элемент | Анимация |
|---------|----------|
| Кнопка «Кубик» (везде) | 3D-«поп» (сжатие/выброс) + вращение эмодзи-кубика + разлёт конфетти из точки клика |
| Пункты меню в хэдере | Материальный ripple-всплеск из точки клика |
| Кнопка «Войти»-стиль (акцентные CTA) | Постоянный светлый блик, пробегающий по кнопке |
| Фильтр-пилюля (бюджет) | Покачивание (wiggle) при переключении состояния |
| Чипы городов/категорий | Покачивание + анимированная галочка при выборе |
| Сердечко «избранное» | Пружинистый поп с поворотом + мини-фонтан сердечек при лайке |
| Карточка места | Тактильное 3D-нажатие при клике |
| Маркер на карте | Подпрыгивание + мини-вспышка искр при клике |
| Фон целиком | Медленно падающие полупрозрачные сердечки (декоративный слой, не мешает контенту) |

## 8. Форма «Предложить место»

Модальное окно (не отдельная страница) с полями:

- локация — переключатель **«По адресу» / «По координатам»** (пользователь заполняет один из двух вариантов, оба равнозначны для системы — см. `06_Domain_Model.md`, Place);
- фотографии — до 3, необязательны (BR-22);
- описание — до 1500 символов (BR-05/BR-21);
- кнопка «Предложить место» и крестик закрытия.

После отправки — крупный всплеск конфетти и сообщение-подтверждение поверх формы, которое само закрывается вместе с окном. Место публикуется сразу (постмодерация, BR-01).

## 9. Куки-согласие

Баннер снизу экрана, появляется через ~0.5 сек после захода на сайт, если согласие ещё не зафиксировано. Одна кнопка — «Подтверждаю». По нажатию в браузере реально сохраняется cookie-флаг согласия (срок — 1 год); при повторном визите баннер не показывается повторно. См. `02_PRD.md` (раздел 3.3, NFR по согласиям) и `05_Business_Rules.md` (правило о куки).

## 10. Секретный вход администратора

Явной кнопки/ссылки «Войти» на сайте нет — гостям входить некуда (регистрации нет, см. ADR-004). Доступ администратора скрыт: 5 быстрых кликов по логотипу «Свиданка» открывают форму ввода пароля. Обоснование и альтернативы — в новом `docs/adr/ADR-005-admin-access.md`.

## 11. Админ-панель

Открывается на весь экран после входа, разделена на вкладки:

1. **Обзор** — метрики (всего мест, новые/постмодерация, скрытые, в архиве, активные города/категории), поясняющий текст про постмодерацию.
2. **Места** — список со статусами (опубликовано/скрыто/архив), действия «Скрыть/Восстановить», «Разархивировать»/«Удалить навсегда» для архивных (BR-16, BR-17), форма прямого добавления места (FR-19).
3. **Отзывы** — «Скрыть/Восстановить» (BR-01–BR-03).
4. **Справочники** — города и категории: добавление, переключатель активности, удаление. Деактивация автоматически переводит привязанные места в архив (BR-16) — проверено в прототипе.
5. **Feature flags** — переключатели на функциональность сайта (см. раздел 12); отключение сразу скрывает соответствующий элемент на публичной части.

Все действия в админ-панели меняют общий источник данных, из которого рендерится и публичная часть сайта — это ключевое архитектурное требование к реальной реализации (не два независимых состояния, а одна модель, см. `07_Database_Design.md`, когда будет готов).

## 12. Feature flags

Новый функциональный блок, не описанный в исходном PRD до этапа прототипирования — добавлен по итогам дизайна и вынесен в `02_PRD.md` (FR-22) и `06_Domain_Model.md` (сущность FeatureFlag). Переключатели в прототипе:

- Фильтр по бюджету
- Фильтр по району
- Фильтр по расстоянию
- Фильтр по категориям
- Режим «Кубик»
- Отзывы (рейтинг на карточках)
- Предложение мест гостями

Смысл: возможность быстро отключить часть функциональности без деплоя кода (например, если фича ведёт себя нестабильно или временно не нужна).
